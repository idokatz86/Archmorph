## Business-Critical Items Identified

The following items represent **gaps that could impact revenue, reliability, or enterprise readiness**. Each should be tracked as a GitHub issue.

### 🔴 P0 — Launch Blockers

#### 1. Disaster Recovery Plan Missing

**Impact:** Enterprise customers will require DR documentation during procurement. A single-region outage could mean total service loss with no documented recovery path.

- No DR runbook exists (only a skeleton `infra/dr/main.tf`)
- No RTO/RPO targets defined
- No automated backup/restore procedures
- No multi-region failover configuration
- No DR drill schedule

**Recommended labels:** `infrastructure`, `security`, `priority:P1`
**Assignee:** DevOps Master, Cloud Master

---

#### 2. Usage Metering Data Stored In-Memory — Data Loss Risk

**Impact:** All usage tracking (analyses, IaC downloads, HLD generations, etc.) is stored in Python dictionaries. Any container restart loses all metering data, breaking billing accuracy and quota enforcement.

**Evidence:** `backend/services/usage_metering.py` uses in-memory dicts keyed by `org_id:YYYY-MM`.

**Fix needed:** Persist usage records to PostgreSQL or Redis with write-through caching.

**Recommended labels:** `backend`, `billing`, `priority:P1`
**Assignee:** Backend Master

---

#### 3. Billing Subscription State Stored In-Memory

**Impact:** Stripe subscription status and customer mappings are held in Python dicts. Container restart loses all subscription state, potentially downgrading paying customers to free tier.

**Evidence:** `backend/routers/billing.py` stores subscription data in `_subscriptions` and `_customers` dicts.

**Fix needed:** Persist to PostgreSQL; use Stripe as source of truth with local cache.

**Recommended labels:** `backend`, `billing`, `priority:P1`
**Assignee:** Backend Master

---

### 🟡 P1 — Revenue & Growth

#### 4. Azure Marketplace Backend Implementation Incomplete

**Impact:** GTM plan targets Azure Marketplace listing (Phase 3–4). Router endpoints exist but delegate to a `marketplace` module that may not implement actual Azure Marketplace SaaS API calls.

**Evidence:** `backend/routers/marketplace.py` has full endpoint structure but the backend service layer needs validation against Azure Marketplace SaaS Fulfillment APIs v2.

**Recommended labels:** `backend`, `marketplace`, `priority:P2`
**Assignee:** Backend Master, API Master

---

#### 5. Load Testing Not Integrated into CI Pipeline

**Impact:** Performance regressions could ship to production undetected. Load test scripts exist (`tests/load_test.py`) but are not wired into any CI workflow.

**Fix needed:** Add a periodic (weekly or pre-release) CI job that runs load tests against a staging environment and gates on latency/error-rate thresholds.

**Recommended labels:** `CI/CD`, `performance`, `priority:P2`
**Assignee:** DevOps Master, Performance Master

---

#### 6. No API Client SDK for Developer Adoption

**Impact:** Enterprise customers and partners need client libraries for integration. Currently, consumers must hand-write HTTP calls against the REST API.

**Fix needed:** Generate OpenAPI spec from FastAPI (auto-generated at `/docs`), then produce TypeScript and Python SDK packages using openapi-generator or similar tooling. Publish to npm/PyPI.

**Recommended labels:** `api`, `enhancement`, `priority:P3`
**Assignee:** API Master, PM Master

---

### 🟢 P2 — Operational Excellence

#### 7. Session Store Failover Strategy

**Current state:** Session store supports both in-memory and Redis backends (`backend/services/session_store.py`). However, there is no documented failover strategy if Redis becomes unavailable in production.

**Recommendation:** Implement circuit-breaker pattern with automatic fallback to in-memory + alert. Document in operational runbook.

**Recommended labels:** `backend`, `infrastructure`, `priority:P3`
